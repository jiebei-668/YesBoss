### 1. 事件接收器：`WebhookController`

这是直接暴露给外部 HTTP 请求的 API 接口，负责捕获办公软件的群聊事件及 @ 事件。

**【关键设计】异步化响应机制：**

企业级 IM（飞书/Slack）的 Webhook 回调有严格的 ACK 超时限制（通常 3 秒内必须返回 200 OK）。如果在接收端同步拉起大模型推理（通常 5-30 秒），必然导致回调超时和重试风暴。

因此，WebhookController 必须实现真正的异步化：
1. 收到 payload 后，仅进行快速验签和解析
2. 将其封装为内部事件并丢入异步线程池（虚拟线程）
3. **当前 HTTP 请求必须立即返回 200 OK**
4. 所有的路由、SessionManager 调用、LLM 推理均在后台线程中异步执行

```java
package com.yesboss.module1.gateway;

import com.yesboss.module1.event.ImWebhookEvent;

/**
 * 统一 Webhook 路由控制器（异步化设计）
 *
 * 【核心约束】接收到 payload 后，必须立即返回 200 OK，
 * 所有耗时操作必须在后台虚拟线程中异步执行。
 */
public interface WebhookController {

    /**
     * 处理飞书 (Feishu) 推送的各类事件
     *
     * 执行流程：
     * 1. 验证飞书签名（快速，<100ms）
     * 2. 解析 payload 为 ImWebhookEvent
     * 3. 丢入 WebhookEventExecutor 异步线程池
     * 4. **立即返回 HTTP 200 OK**（必须在 3 秒内完成）
     *
     * @param payload 飞书推过来的 JSON 报文
     * @return HTTP 200 OK（立即返回，不等待后续处理）
     */
    String handleFeishuEvent(String payload);

    /**
     * 处理 Slack 推送的 Event API 回调
     *
     * 执行流程：
     * 1. 验证 Slack 签名（快速，<100ms）
     * 2. 解析 payload 为 ImWebhookEvent
     * 3. 丢入 WebhookEventExecutor 异步线程池
     * 4. **立即返回 HTTP 200 OK**（必须在 3 秒内完成）
     *
     * @param payload Slack 推过来的 JSON 报文
     * @return HTTP 200 OK（立即返回，不等待后续处理）
     */
    String handleSlackEvent(String payload);

    /**
     * (可选) 终端 CLI 的入口，方便我们在本地用 Ubuntu 测试
     * CLI 模式不需要异步化，可直接同步执行
     * @param command 命令行输入的指令
     */
    void handleCliCommand(String command);
}

/**
 * IM Webhook 内部事件载体
 */
public record ImWebhookEvent(
    String imType,           // FEISHU, SLACK
    String eventType,        // message, group_join, group_delete
    String imGroupId,        // 群聊 ID
    String userId,           // 用户 ID
    String payload,          // 原始 JSON
    long receivedAt          // 接收时间戳
) {}

/**
 * Webhook 异步事件执行器
 * 使用虚拟线程（Virtual Thread）处理后续路由和 LLM 推理
 */
public interface WebhookEventExecutor {

    /**
     * 在后台虚拟线程中异步处理 Webhook 事件
     *
     * 执行流程：
     * 1. 调用 SessionManager 绑定或创建会话
     * 2. 调用 MasterRunner.run() 启动 LLM 推理
     * 3. 通过 IMMessagePusher 主动推送结果
     *
     * @param event 内部事件对象
     */
    void processAsync(ImWebhookEvent event);
}
```

### 2. 路由与生命周期管理：`SessionManager`

系统规定“一次完整的任务即为一次会话”，并且与“新建一个专属群聊”强绑定 。这个组件就是维护外部群聊 ID 和内部 Task ID 映射关系的桥梁。

```java
package com.yesboss.module1.session;

/**
 * 会话生命周期与路由管理器
 * 维护“群聊 ID+渠道类型”与“内部 Task ID”的联合绑定关系。
 */
public interface SessionManager {

    /**
     * 建立路由锚点：当用户在群内下达新任务时，创建或获取绑定的内部 Master Task ID
     * @param imType 接入渠道 (如 FEISHU, SLACK)
     * @param imGroupId 外部办公软件的真实群聊 ID
     * @param topic 任务主题
     * @return 内部全局唯一的 Task Session ID
     */
    String bindOrCreateTaskSession(String imType, String imGroupId, String topic);

    /**
     * 精准路由：通过外部群聊 ID 找到对应的内部任务 ID (高频查询)
     */
    String getInternalTaskId(String imType, String imGroupId);

    /**
     * 物理销毁：处理群聊解散事件
     * [cite_start]当用户删除该群聊时，触发级联数据物理销毁 [cite: 1]。
     * @param imGroupId 解散的群聊 ID
     */
    void destroySessionCascade(String imGroupId);
}

```

### 3. 用户体验包装：`UICardRenderer`

根据需求，系统要追求“结果导向的极简体验” 。这个组件负责把系统内部复杂的推演流和多轮对话，转化为对用户友好的 UI 组件。

```java
package com.yesboss.module1.ui;

/**
 * 极简 UI 渲染器
 * 将内部生硬的执行日志，组装成高级“进度条卡片”或“折叠总结卡片”。
 */
public interface UICardRenderer {

    /**
     * 渲染执行中状态 (极简进度条)
     * [cite_start]群聊内仅显示极其简短的进度条（仅提示“正在完成 xxx 任务...”） [cite: 1]。
     * @param currentTaskName 当前正在执行的子任务名称
     * @return 特定 IM 渠道的 JSON 卡片对象 (如飞书的 Message Card)
     */
    Object renderProgressBar(String currentTaskName);

    /**
     * 渲染拦截挂起状态 (人机回环求助)
     * [cite_start]当遇到敏感操作时，生成包含高危操作详情的卡片，并在群内 @ 用户 [cite: 1]。
     * @param interceptedCommand 被黑名单拦截的高危命令
     * @return 带有“授权/拒绝”按钮的交互式卡片对象
     */
    Object renderSuspensionCard(String interceptedCommand);

    /**
     * 渲染执行后状态 (任务总结卡片)
     * [cite_start]展示各子任务的完成情况，并将中间对话收纳在卡片的折叠面板中 [cite: 1]。
     * @param masterSessionId Master 的会话 ID (用于提取各子任务的报告)
     * @return 带有折叠面板的复杂卡片对象
     */
    Object renderSummaryCard(String masterSessionId);
}

```
### 4. 主动消息推送器：`IMMessagePusher` (新增补丁)

这是为了解决“拦截挂起时主动触达用户”而新增的输出通道。与 `WebhookController`（被动接收）形成闭环，充当系统主动向外伸出的“手”。

```java
package com.yesboss.module1.gateway;

/**
 * IM 主动消息推送器
 * 负责将渲染好的 UI 组件（如挂起求助卡片）或文本，主动推送到指定的外部 IM 群聊中。
 */
 public interface IMMessagePusher {

    /**
     * 将复杂交互卡片推送到指定的群聊
     * +     * @param imType 接入渠道 (如 FEISHU, SLACK)
     * @param imGroupId 外部办公软件的真实群聊 ID (路由锚点)
     * @param uiCardRendererOutput 由 UICardRenderer 渲染出的原生卡片对象
     */
    void pushCardMessage(String imType, String imGroupId, Object uiCardRendererOutput);

    /**
     * 推送纯文本消息 (可选辅助方法，用于简短的进度通告)
     */
    void pushTextMessage(String imType, String imGroupId, String text);
     }
```
