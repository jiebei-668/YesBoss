### 1. 接入与路由模块 (Gateway & Router Module)

* **核心定位：** 系统的“前台接待”。处理所有外部网络请求、多渠道分发和用户界面渲染。
* **包含的核心组件：**
* `WebhookController`: 统一接收飞书/Slack 的 HTTP 回调。
* `SessionManager`: 维护“群聊 ID+渠道类型”与“内部 Task ID”的联合绑定关系，处理建群/解散事件，确保消息精准路由。
* `UICardRenderer`: 把内部生硬的执行日志和工具调用状态，组装成飞书/Slack 原生的高级“进度条卡片”或“折叠总结卡片”。
* **`IMMessagePusher` (新增):** 主动消息推送器。当底层触发拦截时，负责越过大模型直接将求助卡片推送至用户的物理群聊。



### 2. 状态机与调度引擎模块 (State Machine & Orchestrator Module)

* **核心定位：** 系统的“中央处理器”。实现我们在架构里说的“混合驱动模式”，控制 Agent 的生死和流转。
* **包含的核心组件：**
* `MasterRunner` & `WorkerRunner`: 封装 Agent 的动态 ReAct 循环。
* `TaskManager`: 维护任务的状态机流转（PLANNING, RUNNING, SUSPENDED, COMPLETED），并管理父子任务的生命周期。
* `CircuitBreaker`: 20 轮死循环熔断计数器。
* `SuspendResumeEngine`: 负责在触发黑名单时挂起任务，**并跨 Agent 在 Master 全局流中伪造提问上下文**；在接收用户授权后，**依据拦截时的 tool_call_id 注入局部 ToolResult 并**重新拉起上下文。


### 3. 双流上下文管理模块 (Dual-Stream Context Module)

* **核心定位：** 系统的“海马体”。负责记忆的严格隔离、序号排序（防乱序幻觉）与组装。
* **包含的核心组件：**
* `GlobalStreamManager`: 维护 Master 的高维全局探索记忆与人类对话。
* `LocalStreamManager`: 维护 Worker 的局部底层代码与工具试错记忆。
* `InjectionEngine` (固定流程): 负责向下拼装模板，结合 Master 的 `execution_plan`，生成 Worker 的 System Prompt。
* `CondensationEngine` (固定流程): 负责监控 `token_count`，向上调用快速 LLM，把长对话压缩为精简摘要。



### 4. 统一工具链与沙箱模块 (Toolchain & Sandbox Module)

* **核心定位：** 系统的“手脚与护具”。统一管理所有能力并实施安全拦截与因果追踪。
* **包含的核心组件：**
* `ToolRegistry`: 统一工具注册中心。
* `AgentTool` (接口) & `McpToolAdapter`: 将所有原生函数和外部 MCP 服务统一适配为标准工具接口。
* `SandboxInterceptor`: 拦截器，在 `Tool.execute()` 执行前校验黑名单。若拦截，向调度引擎抛出**携带大模型 tool_call_id 的**挂起异常。
*
* `ToolCallTracker` (新增):** 负责维护大模型生成的 `tool_call_id` 与实际执行工具的绑定关系，确保在拦截日志中可精准逆向追溯 Prompt。



### 5. LLM 路由网关模块 (LLM Gateway Module)

* **核心定位：** 系统的“防腐层与翻译官”。屏蔽各种大模型厂商的 API 差异，并处理复杂 SDK 对象的多态映射。
* **包含的核心组件：**
* `LlmClient` (统一接口): 封装核心的聊天与推理调用逻辑。
* `ModelRouter`: 根据任务类型动态分配模型（Master 用大模型，Worker 和摘要用同一模型）。
* **`UnifiedMessage` (新增):** 系统内部定义的标准消息模型，屏蔽一切外部 SDK 细节。
* **`VendorSdkAdapter` (新增防腐层):** 例如 `ClaudeAdapter`。负责将底层数据库中带 `payload_format` 标记的 JSON 字符串，精准反序列化为特定厂商（如 Anthropic）的复杂 Java SDK 对象（如 `TextBlock`, `ToolUseBlock`），确保 `Resume` 唤醒时数据结构绝对完整。



### 6. 异步持久化模块 (Async Persistence Module)

* **核心定位：** 系统的"黑匣子"。解决 SQLite 高并发写入痛点，提供极其健壮的数据底座。支持完整的 CRUD 操作，包括级联删除（用于群聊解散时的数据物理销毁）。
* **包含的核心组件：**
* `DbWriteEvent` (新增: 写事件契约): 系统的事件基类。利用事件溯源思想，将所有写库动作封装为多态事件对象（如 InsertMessageEvent, UpdateTaskStatusEvent, DeleteMessagesEvent, DeleteTaskSessionEvent），彻底解耦业务层与持久层。
* `SingleThreadDbWriter`: (包含内置的 MemoryWriteQueue): 系统的后台写库引擎。它内部封装了一个基于内存的无锁阻塞队列（LinkedBlockingQueue）作为缓冲池。对外只暴露 submit(event) 方法；内部启动全局唯一的守护线程，不断从私有队列拉取事件并顺序执行 SQL，从物理层面上彻底消灭并发写冲突。
* `Repositories` (如 TaskSessionRepo, ChatMessageRepo): 面向业务层的 DAO 接口。读操作直接并发查库；写操作全部拦截，包装成 DbWriteEvent 后丢进 MemoryWriteQueue。
