### 1. 基础结构：流类型定义

首先，为了防止在代码中传错字符串，我们需要把“双流”的标识固化下来，这直接对应了数据库 `chat_message` 表中的 `stream_type` 字段。

```java
package com.yesboss.module3.context;

/**
 * 记忆流类型枚举
 */
public enum StreamType {
    GLOBAL, // 全局流：仅存人类对话与 Master 的高维抽象
    LOCAL   // 局部流：存满底层代码与 Worker 的工具试错噪音
}

```

### 2. 双流管理器：`GlobalStreamManager` & `LocalStreamManager`

这两个管理器对上层暴露，屏蔽了底层的序号排序（`sequence_num`）生成和数据库异步写入逻辑。

```java
package com.yesboss.module3.context;

import com.yesboss.module5.model.UnifiedMessage;
import java.util.List;

/**
 * 全局流管理器 (Global Stream Manager)
 * 专供 Master 和人类交互使用，绝对隔离底层执行的脏数据。
 */
public interface GlobalStreamManager {

    /**
     * 追加用户（人类）发来的消息
     * 内部会自动生成 sequence_num，并封装为 DbWriteEvent 丢给异步队列。
     * @param masterSessionId Master 的会话 ID
     * @param humanText 人类输入的文本
     */
    void appendHumanMessage(String masterSessionId, String humanText);

    /**
     * 追加 Master 的回复或规划思考
     * @param masterSessionId Master 的会话 ID
     * @param masterMessage Master 产出的 UnifiedMessage
     */
    void appendMasterMessage(String masterSessionId, UnifiedMessage masterMessage);

    /**
     * 获取 Master 的完整上下文，用于下一次向大模型发起请求
     * @param masterSessionId Master 的会话 ID
     * @return 严格按 sequence_num 排序的 UnifiedMessage 列表
     */
    List<UnifiedMessage> fetchContext(String masterSessionId);

    /**
     * 向 Master 的全局流插入系统生成的消息（非人类输入，也非 LLM 生成）
     * 用于人机回环、框架级通知等跨 Agent 伪造场景。
     *
     * **核心使用场景：**
     * 当 Worker 触发黑名单拦截时，系统需要越过 Master 的 LLM，
     * 直接在 Master 的全局流中插入"系统提问"消息，然后通过 IM 推送给用户。
     *
     * @param masterSessionId Master 的会话 ID
     * @param systemText 系统生成的文本内容
     */
    void appendSystemMessage(String masterSessionId, String systemText);
}


/**
 * 局部流管理器 (Local Stream Manager)
 * 专供 Worker 底层试错使用，各 Worker 之间物理隔离，绝不交叉。
 */
public interface LocalStreamManager {

    /**
     * 追加 Worker 的执行思考或工具调用指令
     * @param workerSessionId Worker 的会话 ID
     * @param workerMessage Worker 产出的 UnifiedMessage
     */
    void appendWorkerMessage(String workerSessionId, UnifiedMessage workerMessage);

    /**
     * 追加底层沙箱工具的执行结果（报错信息或成功输出）
     * @param workerSessionId Worker 的会话 ID
     * @param toolResult 系统执行完工具后返回的结果封装
     */
    void appendToolResult(String workerSessionId, UnifiedMessage toolResult);

    /**
     * 获取 Worker 的局部上下文
     * @param workerSessionId Worker 的会话 ID
     * @return 严格按 sequence_num 排序的局部历史记录
     */
    List<UnifiedMessage> fetchContext(String workerSessionId);
    
    /**
     * 获取当前局部流的总 Token 消耗估算
     * 供 CondensationEngine 的压缩探针使用
     */
    int getCurrentTokenCount(String workerSessionId);
}

```

### 3. 跨 Agent 状态交接引擎（框架固定流程）

根据架构设计，当任务在 Master 和 Worker 之间交接时，大模型会被剥夺自由度，强制切入底层写死的框架逻辑。这就对应了下面这两个引擎。

```java
package com.yesboss.module3.engine;

import com.yesboss.module5.model.UnifiedMessage;

/**
 * 向下注入引擎 (Top-Down Injection Engine)
 * 负责在 Master 拆分出子任务后，组装 Worker 的初始记忆。
 */
public interface InjectionEngine {

    /**
     * 拼装 Worker 的初始 System Prompt
     * 将提取全局流中的规则，通过硬编码模板与任务目标拼装，作为初始系统提示词静默注入给新建的 Worker。
     *
     * @param masterSessionId 父任务的 Session ID (用于提取全局规则)
     * @param assignedTask 给当前 Worker 下发的死命令 (如："重构 tui 模块")
     * @return 生成的标准化系统提示词消息，随后应由 LocalStreamManager 存入库中
     */
    UnifiedMessage injectInitialContext(String masterSessionId, String assignedTask);
}


/**
 * 向上收敛引擎 (Bottom-Up Condensation Engine)
 * 负责解决上下文超载，提纯脏数据。
 */
public interface CondensationEngine {

    /**
     * 触发局部流压缩并合并回全局流
     * 开启独立的全新 Context，调用和 Worker 相同的模型，注入"系统级日志压缩规则"，
     * 将庞大的局部流历史提纯为简短的执行报告，并最终合并回 Master 的全局流。
     *
     * @param workerSessionId 触发熔断或顺利完成的 Worker Session ID
     * @param masterSessionId 需要接收汇报的 Master Session ID
     * @return 压缩后的精简摘要文本
     */
    String condenseAndMergeUpwards(String workerSessionId, String masterSessionId);
    
    /**
     * Token 超载探针拦截
     * [cite_start]检查当前 Worker 的局部流 Token 是否超出阈值。如果是，可以在内部触发会话内的 Compact 机制 [cite: 1]。
     */
    boolean checkAndCompactIfNeeded(String workerSessionId);
}
```
