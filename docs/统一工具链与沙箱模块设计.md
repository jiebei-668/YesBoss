### 1. 统一工具抽象接口：`AgentTool`

这是所有工具的终极契约。不管是写本地文件的原生函数，还是通过 MCP 协议连过来的外部服务，在系统内部全长这个样子。

```java
package com.yesboss.module4.tool;

/**
 * 统一工具抽象接口
 * 所有的原生能力与外部 MCP 服务在注册入网时，均被包装为此标准接口。
 */
public interface AgentTool {

    /**
     * 工具的唯一标识名称 (供 LLM 调用，如 "write_file", "mcp_list_dir")
     */
    String getName();

    /**
     * 工具的详细描述 (极其重要，直接影响 LLM 的 Prompt 理解)
     */
    String getDescription();

    /**
     * 工具入参的 JSON Schema 定义
     * 用于向大模型声明该工具需要什么参数 (如 Claude 的 input_schema)
     */
    String getParametersJsonSchema();

    /**
     * 核心执行逻辑
     * @param argumentsJson LLM 生成并传入的参数 JSON 字符串
     * @return 工具执行的真实结果，或者是底层的异常报错堆栈
     */
    String execute(String argumentsJson) throws Exception;

    /**
     * 绕过沙箱检查的执行逻辑（专用于人机回环用户授权后）
     *
     * **核心场景：**
     * 当 Worker 触发黑名单拦截，用户明确授权后，系统需要绕过 SandboxInterceptor，
     * 直接执行该工具。此方法仅在被 SuspendResumeEngine.resume() 调用时使用。
     *
     * @param argumentsJson LLM 生成并传入的参数 JSON 字符串
     * @return 工具执行的真实结果，或者是底层的异常报错堆栈
     */
    String executeWithBypass(String argumentsJson) throws Exception;
}

```

### 2. 工具注册与分级管控：`ToolRegistry`

在主从架构中，Master 和 Worker 的权限边界是完全不同的。注册中心不仅要管工具的注册，还要充当 RBAC（基于角色的访问控制）的角色。

```java
package com.yesboss.module4.registry;

import com.yesboss.module4.tool.AgentTool;
import java.util.List;

/**
 * 统一工具注册中心
 * 统一管理原生 Tool 与 MCP 适配器 Tool。
 */
public interface ToolRegistry {

    /**
     * 注册一个新的工具到系统中
     */
    void register(AgentTool tool);

    /**
     * 根据名称精确获取工具实例
     */
    AgentTool getTool(String toolName);

    /**
     * 获取指定角色被授权使用的工具列表 (分级权限控制)
     * * Master 权限边界：仅被授权调用只读/探索型工具（如查看目录）。
     * Worker 权限边界：被授权调用读写执行全量工具（受严格沙箱管控）。
     * * @param agentRole 枚举值：MASTER 或 WORKER
     * @return 当前角色可用的工具集合，用于组装发给大模型的 Prompt
     */
    List<AgentTool> getAvailableTools(String agentRole);
}

```

### 3. 安全护具：`SandboxInterceptor` 与异常定义

这是防止 Agent“暴走”的关键防线。当 Worker 尝试越权操作时，拦截器必须立刻阻断执行并向上层调度引擎抛出挂起信号。

```java
package com.yesboss.module4.sandbox;

/**
 * 高危命令安全挂起异常
 * 这是一个业务异常，当触发拦截时，立刻向上抛出，由状态机引擎接管并进入 SUSPENDED 状态。
 */
public class SuspendExecutionException extends RuntimeException {
    private final String interceptedCommand;
    private final String toolCallId; // 新增：必须携带引发拦截的因果纽带 ID 
    
    public SuspendExecutionException(String interceptedCommand, String toolCallId) {
        super("操作触发安全沙箱黑名单，执行已挂起。");
        this.interceptedCommand = interceptedCommand;
        this.toolCallId= toolCallId;
    }
    
    public String getInterceptedCommand() { return interceptedCommand; }
    public String getToolCallId() { return toolCallId; } // 新增 getter
}

/**
 * 安全沙箱拦截器 (拦截器逻辑核心)
 *
 * **拦截后的强制流程：** 抛出 SuspendExecutionException 后，系统将触发挂起流程。
 * 若是 Worker 触发，必须通过 Master 向用户确认，严禁 Master 自行决策。
 */
public interface SandboxInterceptor {

    /**
     * 执行前置校验 (Pre-check)
     * 在 Tool.execute() 执行前显式调用以校验黑名单。
     *
     * @param tool 准备调用的工具实例
     * @param argumentsJson LLM 传入的执行参数
     * @param toolCallId 当前触发调用的 ID，用于透传给挂起引擎
     * @throws SuspendExecutionException 若命中黑名单则阻断，并携带 toolCallId 抛出
     *
     * **抛出此异常后的处理流程：**
     * 1. SuspendResumeEngine 捕获异常并将任务状态转为 SUSPENDED
     * 2. 若是 Worker 触发，通知其父级 Master
     * 3. Master 通过 IM 在群内 @ 用户展示拦截详情
     * 4. 系统无限期等待用户决策（授权/拒绝/补充指令）
     * 5. 只有用户本人可恢复执行，Master 严禁自行决策
     */
    void preCheck(AgentTool tool, String argumentsJson, String toolCallId) throws SuspendExecutionException;

    /**
     * 检查工具名称是否在黑名单中
     *
     * **细粒度检查方法**：单独检查工具名层面是否被禁止。
     * 某些高危工具（如系统级格式化工具）可能直接在工具名层面就被禁止。
     *
     * @param toolName 工具的唯一标识名称
     * @return true 如果工具名在黑名单中，false 否则
     */
    boolean checkBlacklist(String toolName);

    /**
     * 检查工具调用参数是否包含危险命令
     *
     * **细粒度检查方法**：单独检查参数内容是否触发黑名单规则。
     * 即使工具本身是安全的，其参数可能包含危险操作（如 rm -rf、|bash 等）。
     *
     * **支持的检查模式：**
     * - 精确匹配：如 "rm -rf /home"
     * - 正则表达式：如 "curl .*\\|bash"、"eval .*"
     *
     * @param argumentsJson 工具调用参数的 JSON 字符串
     * @return true 如果参数匹配黑名单规则，false 否则
     */
    boolean checkArguments(String argumentsJson);
}

```

#### 实现示例：`SandboxInterceptorImpl`

以下展示如何使用细粒度检查方法来实现 `preCheck()`：

```java
package com.yesboss.module4.sandbox.impl;

import com.yesboss.module4.sandbox.SandboxInterceptor;
import com.yesboss.module4.sandbox.SuspendExecutionException;
import com.yesboss.module4.tool.AgentTool;
import java.util.List;
import java.util.regex.Pattern;

/**
 * 安全沙箱拦截器实现
 */
public class SandboxInterceptorImpl implements SandboxInterceptor {

    // 工具名黑名单（直接禁止的工具）
    private static final List<String> TOOL_NAME_BLACKLIST = List.of(
        "format_disk",
        "wipe_partition"
    );

    // 参数黑名单（正则表达式）
    private static final List<Pattern> ARGUMENT_BLACKLIST = List.of(
        Pattern.compile("rm\\s+-rf\\s+.*"),         // rm -rf ...
        Pattern.compile("curl\\s+.*\\|\\s*(bash|sh)"),  // curl ... | bash/sh
        Pattern.compile("wget\\s+.*\\|\\s*(bash|sh)"),  // wget ... | bash/sh
        Pattern.compile("eval\\s+.*"),              // eval ...
        Pattern.compile("exec\\s+.*"),              // exec ...
        Pattern.compile(">\\s*/etc/(passwd|shadow|sudoers)") // 写入系统敏感文件
    );

    @Override
    public void preCheck(AgentTool tool, String argumentsJson, String toolCallId) throws SuspendExecutionException {
        // 第一步：检查工具名是否在黑名单
        if (checkBlacklist(tool.getName())) {
            throw new SuspendExecutionException(
                "工具 [" + tool.getName() + "] 在黑名单中",
                toolCallId
            );
        }

        // 第二步：检查参数是否包含危险命令
        if (checkArguments(argumentsJson)) {
            throw new SuspendExecutionException(
                "参数 [" + argumentsJson + "] 触发黑名单规则",
                toolCallId
            );
        }

        // 两者都通过，校验放行
    }

    @Override
    public boolean checkBlacklist(String toolName) {
        return TOOL_NAME_BLACKLIST.contains(toolName);
    }

    @Override
    public boolean checkArguments(String argumentsJson) {
        // 检查参数是否匹配任何一个黑名单正则表达式
        for (Pattern pattern : ARGUMENT_BLACKLIST) {
            if (pattern.matcher(argumentsJson).find()) {
                return true;
            }
        }
        return false;
    }
}
```

**使用场景说明：**

1. **preCheck()**：由 `WorkerRunner` 在工具执行前统一调用，是主要的拦截入口
2. **checkBlacklist()**：可独立使用，用于 UI 层展示哪些工具被禁用，或用于工具注册时的预检
3. **checkArguments()**：可独立使用，用于 IDE 插件或调试工具实时校验参数安全性

### 4. 审计黑匣子：`ToolCallTracker`

根据设计文档的强调，大模型生成的 `tool_call_id` 是因果追踪的绝对纽带。这个追踪器需要把执行明细封转为底层事件。

```java
package com.yesboss.module4.tracker;

/**
 * 工具调用追踪器 (审计日志)
 * 负责维护大模型生成的 tool_call_id 与实际执行工具的绑定关系。
 */
public interface ToolCallTracker {

    /**
     * 记录一次完整的工具调用流水
     * 该方法内部不应直接写库，而是将数据封装为 InsertToolExecutionLogEvent，
     * 投递给 DbWriteEvent 异步内存无锁队列。
     *
     * @param sessionId 任务会话 ID
     * @param toolCallId 大模型生成的因果纽带 ID
     * @param toolName 工具接口名
     * @param argumentsJson 参数 JSON
     * @param result 真实执行结果
     * @param isIntercepted 沙箱阀门标记：1 = 拦截挂起，0 = 安全放行
     */
    void trackExecution(String sessionId, String toolCallId, String toolName, String argumentsJson, String result, boolean isIntercepted);
}

```

### 设计总结与联动演练

有了这套接口，再结合前面设计的模块 2 和模块 3，Worker 在底层 ReAct 循环中的执行伪代码看起来会极其优雅，完全印证了主从架构的核心逻辑 ：

```java
// WorkerRunner 内部调用工具的伪代码推演
AgentTool tool = toolRegistry.getTool(toolCall.name());

try {
    // 1. 沙箱前置校验：护具生效
    sandboxInterceptor.preCheck(tool, toolCall.argumentsJson(), toolCall.id());
    
    // 2. 真实执行：手脚干活
    String result = tool.execute(toolCall.argumentsJson());
    
    // 3. 记录日志：黑匣子留档 (isIntercepted = false)
    toolCallTracker.trackExecution(sessionId, toolCall.id(), tool.getName(), toolCall.argumentsJson(), result, false);
    
    // 4. 返回结果给海马体
    localStreamManager.appendToolResult(sessionId, UnifiedMessage.ofToolResult(toolCall.id(), result, false));

} catch (SuspendExecutionException ex) {
    // 触发拦截！
    // 记录拦截日志 (isIntercepted = true)
    toolCallTracker.trackExecution(sessionId, toolCall.id(), tool.getName(), toolCall.argumentsJson(), ex.getMessage(), true);
    
    // 向上抛给 SuspendResumeEngine，等待人类救援...
    throw ex; 
}
```
