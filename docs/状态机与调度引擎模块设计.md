### 1. 状态机基石：枚举与任务中枢 `TaskManager`

首先，我们需要把架构图里提到的那几个核心状态固化下来。`TaskManager` 是对外的唯一操作入口，它负责收口所有的状态流转，并联动我们之前设计好的数据库事件队列。

```java
package com.yesboss.module2.engine;

/**
 * 任务生命周期状态枚举
 */
public enum TaskStatus {
    PLANNING,   // 规划中 (Master 专属)
    RUNNING,    // 正在执行 (Master/Worker 通用)
    SUSPENDED,  // 挂起状态 (触发安全黑名单，等待人类救援)
    COMPLETED,  // 顺利完成
    FAILED      // 彻底失败 (如触发熔断)
}

/**
 * 任务中枢管理器
 * 负责维护任务的状态机流转，并管理父子任务的生命周期。
 */
public interface TaskManager {

    /**
     * 开启一个全新的总任务 (Master 会话)
     * @param imType 渠道类型 (如 FEISHU, SLACK)
     * @param imGroupId 绑定的群聊 ID
     * @param topic 任务主题
     * @return 全局唯一的 Master Session ID
     */
    String createMasterTask(String imType, String imGroupId, String topic);

    /**
     * 拆分并下发一个子任务 (Worker 会话)
     * @param parentSessionId Master 的 Session ID
     * @param assignedTask 给 Worker 下发的明确任务文本
     * @return 全局唯一的 Worker Session ID
     */
    String createWorkerTask(String parentSessionId, String assignedTask);

    /**
     * 核心状态机流转方法
     * 这里内部必须做严格的状态校验 (例如 FAILED 不能直接转 RUNNING)
     */
    void transitionState(String sessionId, TaskStatus newStatus);

    /**
     * 获取当前任务状态
     */
    TaskStatus getStatus(String sessionId);

    /**
     * 获取父级会话ID
     * 用于人机回环流程中，Worker 触发拦截时需要找到其父级 Master，
     * 以便向 Master 的全局流插入系统消息。
     *
     * @param sessionId 当前会话ID（Worker 或 Master）
     * @return 父级会话ID（Master 的 parent 为 null），若当前已是 Master 则返回 null
     */
    String getParentSessionId(String sessionId);

    /**
     * 获取 IM 路由信息
     * 用于人机回环流程中，SuspendResumeEngine 需要通过 Master SessionId
     * 获取对应的 imType 和 imGroupId，以便推送消息到正确的群聊。
     *
     * @param masterSessionId Master 的会话 ID
     * @return 包含 imType 和 imGroupId 的路由对象，或抛出异常如果不存在
     */
    ImRoute getImRoute(String masterSessionId);
}

/**
 * IM 路由信息对象
 * 用于封装会话与外部 IM 系统的绑定关系
 */
public record ImRoute(String imType, String imGroupId) {}
```

### 2. 混合驱动引擎：`AgentRunner` 体系

`MasterRunner` 和 `WorkerRunner` 的本质都是在跑 ReAct (Reason + Act) 循环。我们可以抽象出一个通用的 `AgentRunner` 接口。

```java
package com.yesboss.module2.runner;

/**
 * 智能体执行器统一接口
 * 封装了 Agent 的动态 ReAct 循环。
 */
public interface AgentRunner {
    
    /**
     * 启动 Agent 的执行循环
     * 建议在底层实现时，将其丢入 Executors.newVirtualThreadPerTaskExecutor() 中运行
     * * @param sessionId 当前任务的 Session ID
     */
    void run(String sessionId);
}

/**
 * 总 Agent 执行器 (Master)
 * 权限边界：只读探索工具、全局流读写。
 */
public interface MasterRunner extends AgentRunner {
    
    /**
     * Master 的专属能力：制定执行蓝图
     * 在环境探索完毕后，生成 JSON 格式的子任务队列。
     */
    String generateExecutionPlan(String sessionId);
}

/**
 * 子 Agent 执行器 (Worker)
 * 权限边界：读写全量工具（受沙箱管控）、局部流读写。
 */
public interface WorkerRunner extends AgentRunner {

    /**
     * Worker 的专属能力：向上汇报
     * 历经多轮尝试完成后，生成简短的执行报告供框架收敛。
     */
    String generateExecutionReport(String sessionId);

    /**
     * 【核心规约】唤醒状态判定
     *
     * **问题背景：**
     * 当 Worker 触发人机回环（黑名单拦截）时，Worker 线程会完全退出释放资源。
     * 用户决策后，SuspendResumeEngine 会创建新线程调用 run(sessionId)。
     * 此时新线程内存中没有任何局部变量，如果直接执行初始化流程会导致重复注入。
     *
     * **强制判定逻辑：**
     * 在 run() 方法入口处，必须先从 LocalStreamManager 拉取完整的局部流历史。
     * 判断最后一条消息的类型：
     *
     * - 如果最后一条消息是 ToolResult（role=TOOL 或包含 ToolResult）：
     *   → 代表这是从 SUSPENDED 状态唤醒
     *   → 必须跳过任务初始化（跳过 InjectionEngine 的 Prompt 注入）
     *   → 直接基于现有上下文继续 ReAct 循环，大模型会看到 ToolResult 并继续推理
     *
     * - 如果最后一条消息不是 ToolResult（或局部流为空）：
     *   → 代表这是新创建的 Worker
     *   → 执行完整的初始化流程（调用 InjectionEngine.injectInitialContext）
     *   → 然后开始正常的 ReAct 循环
     *
     * **伪代码示例：**
     * <pre>
     * void run(String sessionId) {
     *     // 1. 拉取局部流历史
     *     List<UnifiedMessage> context = localStreamManager.fetchContext(sessionId);
     *
     *     // 2. 判定是否为唤醒状态
     *     boolean isResuming = !context.isEmpty()
     *         && context.get(context.size() - 1).role() == Role.TOOL;
     *
     *     if (!isResuming) {
     *         // 新创建的 Worker：执行初始化
     *         UnifiedMessage initialPrompt = injectionEngine.injectInitialContext(...);
     *         localStreamManager.appendWorkerMessage(sessionId, initialPrompt);
     *     }
     *     // 唤醒状态：跳过初始化，直接进入 ReAct 循环
     *
     *     // 3. 开始/继续 ReAct 循环
     *     while (running) {
     *         UnifiedMessage response = llmClient.chat(context, ...);
     *         // ... 处理工具调用
     *     }
     * }
     * </pre>
     *
     * @param sessionId Worker 的会话 ID
     * @return true 如果这是从挂起状态唤醒，false 如果这是新创建的 Worker
     */
    boolean isResumingFromSuspension(String sessionId);
}

```

### 3. 安全与防卡死防线：熔断器与挂起引擎

这部分是保证系统不失控的核心。我们需要限制 Worker 的无脑试错，以及处理高危操作的人机回环。

#### 核心设计决策：Worker 挂起时采用线程完全退出模式

当 Worker 触发黑名单拦截需要人机回环时，Worker 的 ReAct 循环线程会**完全退出**，释放所有内存资源。挂起期间仅保留数据库中的持久化状态，不占用任何线程或内存资源。

**资源占用对比：**

| 资源类型 | 挂起期间占用 | 说明 |
|---------|-------------|------|
| 线程 | ❌ 不占用 | `run()` 方法正常退出，线程结束 |
| 线程栈 | ❌ 不占用 | 随线程一起释放 |
| LocalStream 内存缓存 | ❌ 不占用 | 如有缓存则清空，恢复时从数据库重建 |
| 数据库状态 | ✅ 占用 | `task_session.status = 'SUSPENDED'` |
| 拦截现场 | ✅ 占用 | `toolCallId`, `toolName`, `argumentsJson` 持久化到数据库 |

**挂起流程（suspendForApproval）：**
1. 将任务状态转为 `SUSPENDED`
2. 持久化拦截现场到数据库
3. 发送 IM 通知给用户
4. Worker 的 `run()` 方法捕获异常后**正常退出**
5. 系统进入挂起状态，仅依赖数据库维持

**恢复流程（resume）：**
1. 从数据库读取 `toolCallId`, `toolName`, `argumentsJson`
2. 根据用户决策执行工具或伪造错误
3. 将结果注入 `LocalStreamManager`
4. 创建**新线程**重新调用 `WorkerRunner.run(sessionId)`
5. 新线程从数据库 `fetchContext()` 重建上下文
6. Worker 看到 ToolResult，继续 ReAct 循环

**采用此模式的原因：**
- ✅ 完全避免资源泄漏，即使用户永久不回复
- ✅ 支持服务重启后恢复挂起任务
- ✅ 符合云原生无状态设计理念
- ✅ 可支持大量任务同时挂起（仅受数据库容量限制）

**需要注意：**
- ⚠️ `resume()` 时必须从数据库完整重建上下文
- ⚠️ 恢复速度略慢于阻塞模式（但在可接受范围内）

```java
package com.yesboss.module2.safeguard;

/**
 * 死循环熔断器
 * [cite_start]防止 Agent 在同一个报错上反复重试卡死 [cite: 1]。
 */
public interface CircuitBreaker {

    /**
     * 记录并检查对话轮数
     * @param sessionId 当前的 Session ID
     * [cite_start]@throws CircuitBreakerOpenException 当达到 20 轮上限时抛出熔断异常 [cite: 1]
     */
    void checkAndIncrement(String sessionId);

    /**
     * 重置计数器 (通常在任务彻底完成或人工介入修复后调用)
     */
    void reset(String sessionId);
}

/**
 * 挂起与恢复引擎
 * [cite_start]处理"人机回环 (Human-in-the-loop)"的核心组件 [cite: 1]。
 *
 * **用户确认的强制性原则：**
 * 当 Worker 触发黑名单被拦截时，系统不得由 Master 自行决策恢复。
 * 必须通过 Master 向用户发起确认请求，只有用户本人可通过 IM 授权或拒绝。
 */
public interface SuspendResumeEngine {

    /**
     * 挂起当前任务
     * 当底层 SandboxInterceptor 抛出黑名单拦截异常时调用。
     * [cite_start]它需要将任务状态切为 SUSPENDED，并通过网关在 IM 群内 @ 用户 [cite: 1]。
     *
     * **重要：** 如果是 Worker 触发，必须先通知其父级 Master，由 Master 转发给用户。
     * Master 严禁自行做出决策，必须等待用户通过 IM 的明确回复。
     *
     # @param workerSessionId 触发拦截的 Worker Session ID (明确指代)
     * @param interceptedCommand 被拦截的高危命令明细
     * @param toolCallId 触发拦截的大模型 tool_call_id (补齐缺口：用于记录因果并在唤醒时缝合上下文)
     */
    void suspendForApproval(String workerSessionId, String interceptedCommand, String toolCallId);

    /**
     * 从挂起状态唤醒任务
     * 接收到用户的明确授权或拒绝后，引擎需构造带有用户意图的 ToolResult，
     * 注入回 LocalStreamManager，并重新拉起 Worker 的大模型上下文。
     * @param workerSessionId 被挂起的 Worker Session ID
     * @param toolCallId 需要被响应补齐的工具调用 ID
     * @param isApproved 用户是否同意继续执行
     * @param humanFeedback 用户补充的指导意见 (可选)
     */
    void resume(String workerSessionId, String toolCallId, boolean isApproved, String humanFeedback);
}
```
