### 一、 核心实体表结构详解 (V7.0 终极版)

#### 1. 任务会话表 (`task_session`)

**核心定位：** 状态机引擎。管理 Master/Worker 的生命周期、目标拆解以及外部 IM 的路由映射。

| 字段名 | SQLite 类型 | 约束 | 核心业务含义与带注释说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 全局唯一会话 ID (如: `tsk_9a8b7c`)。 |
| `parent_id` | TEXT | 允许为空 | **主从指针。** 若为空则代表它是总控 Master；若有值则代表它是干活的 Worker。 |
| `im_type` | TEXT | NOT NULL | **接入渠道。** 枚举：`FEISHU` (飞书), `SLACK`, `CLI` (终端)。用于区分不同的 API 客户端。 |
| `im_group_id` | TEXT | NOT NULL | **路由锚点。** 绑定的群聊 ID，配合 `im_type` 唯一确定挂起唤醒时的消息推送目标。 |
| `role` | TEXT | NOT NULL | **Agent 身份。** 枚举：`MASTER`, `WORKER`。便于代码层使用工厂模式实例化不同对象。 |
| `status` | TEXT | NOT NULL | **状态机字典。** `PLANNING` (规划中), `RUNNING`, `SUSPENDED` (被沙箱拦截挂起), `COMPLETED`, `FAILED` (熔断)。 |
| `topic` | TEXT | NOT NULL | **会话标题。** 简短的描述，主要用于前端卡片侧边栏外显。 |
| `execution_plan` | TEXT | 允许为空 | **【Master 专属】规划蓝图。** 存放 Master 规划好的子任务队列 (JSON 数组格式)。 |
| `assigned_task` | TEXT | 允许为空 | **【Worker 专属】执行死命令。** 框架给 Worker 下发的明确任务文本，Worker 终其一生只为完成它。 |
| `created_at` | INTEGER | NOT NULL | 创建时间 (毫秒级时间戳)。 |
| `updated_at` | INTEGER | NOT NULL | 状态最后更新时间。 |

---

#### 2. 上下文消息表 (`chat_message`)

**核心定位：** 双流记忆海马体。支撑大模型 Prompt 拼装、隔离噪音，并兼容复杂 SDK 多态对象的反序列化。

| 字段名 | SQLite 类型 | 约束 | 核心业务含义与带注释说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 消息全局唯一 ID (如: `msg_1x2y3z`)。 |
| `session_id` | TEXT | NOT NULL | 关联 `task_session.id`，实现不同 Worker 间的局部记忆物理隔离。 |
| `stream_type` | TEXT | NOT NULL | **双流标记。** 枚举：`GLOBAL` (仅存人类对话与高维抽象), `LOCAL` (存满底层代码与工具试错噪音)。 |
| `sequence_num` | INTEGER | NOT NULL | **时序防线。** 会话内单调递增的绝对序号。保证组装 Context 给 LLM 时严格按顺序，杜绝幻觉。 |
| `msg_role` | TEXT | NOT NULL | OpenAI/Claude 标准角色：`system`, `user`, `assistant`, `tool`。 |
| **`payload_format`** | **TEXT** | **NOT NULL** | **【序列化契约】** 标识 `content` 的格式。枚举：`PLAIN_TEXT` (纯文本), `ANTHROPIC_BLOCKS` (Claude 多态数组), `OPENAI_CHATS`。 |
| `content` | TEXT | NOT NULL | **多态载荷。** 根据 `payload_format` 解析，可能是纯文本，也可能是复杂的 JSON 序列化字符串（包含 ToolUse 等 Block）。 |
| `token_count` | INTEGER | DEFAULT 0 | **压缩探针。** 记录耗费，一旦超标即触发框架的固定流程：调用 Worker 模型进行摘要收敛。 |
| `created_at` | INTEGER | NOT NULL | 消息落盘的毫秒级时间戳。 |

---

#### 3. 工具执行流水表 (`tool_execution_log`)

**核心定位：** 沙箱审计黑匣子。通过 `tool_call_id` 与大模型推理上下文完美缝合，提供极简 UI 渲染数据源。

| 字段名 | SQLite 类型 | 约束 | 核心业务含义与带注释说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 流水唯一 ID。 |
| `session_id` | TEXT | NOT NULL | 发起调用的 Agent 会话 ID。 |
| **`tool_call_id`** | **TEXT** | **NOT NULL** | **【因果纽带】大模型生成的调用 ID。** 凭此 ID 可去 `chat_message` 精准追溯是哪一次大模型思考 (Thought) 触发了该操作。 |
| `tool_name` | TEXT | NOT NULL | 统一适配后的工具接口名 (如: `write_file`, `mcp_list_dir`)。 |
| `arguments` | TEXT | NOT NULL | LLM 试图传入的参数 JSON 字符串。 |
| `result` | TEXT | 允许为空 | 工具返回的真实执行结果，或者是底层的异常报错堆栈。 |
| `is_intercepted` | INTEGER | DEFAULT 0 | **沙箱阀门。** `1` = 命中黑名单拦截，上层状态机立刻切为 SUSPENDED；`0` = 安全放行。 |
| `created_at` | INTEGER | NOT NULL | 执行时间。 |

---

### 二、 生产级 SQLite 建表语句 (DDL 与 索引)

这段 SQL 包含了极其关键的**复合索引**，直接决定了整个引擎在极高并发下的查询性能：

```sql
-- ==========================================
-- 1. 任务会话表 (task_session)
-- ==========================================
CREATE TABLE task_session (
    id TEXT PRIMARY KEY,
    parent_id TEXT,
    im_type TEXT NOT NULL,
    im_group_id TEXT NOT NULL,
    role TEXT NOT NULL,
    status TEXT NOT NULL,
    topic TEXT NOT NULL,
    execution_plan TEXT,
    assigned_task TEXT,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

-- 索引：加速 Webhook 收到消息时极速定位到具体的归属任务 (最高频查询)
CREATE INDEX idx_session_im_route ON task_session(im_type, im_group_id);
-- 索引：加速查找某个 Master 麾下的所有 Worker 子任务
CREATE INDEX idx_session_parent ON task_session(parent_id);


-- ==========================================
-- 2. 上下文消息表 (chat_message)
-- ==========================================
CREATE TABLE chat_message (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    stream_type TEXT NOT NULL,
    sequence_num INTEGER NOT NULL,
    msg_role TEXT NOT NULL,
    payload_format TEXT NOT NULL,
    content TEXT NOT NULL,
    token_count INTEGER DEFAULT 0,
    created_at INTEGER NOT NULL
);

-- 【最高优复合唯一索引】：保证查询时双流绝对隔离，且严格按序号排序，防乱序导致大模型崩溃
CREATE UNIQUE INDEX idx_chat_msg_seq ON chat_message(session_id, stream_type, sequence_num);


-- ==========================================
-- 3. 工具执行流水表 (tool_execution_log)
-- ==========================================
CREATE TABLE tool_execution_log (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    tool_call_id TEXT NOT NULL,
    tool_name TEXT NOT NULL,
    arguments TEXT NOT NULL,
    result TEXT,
    is_intercepted INTEGER DEFAULT 0,
    created_at INTEGER NOT NULL
);

-- 索引：加速前端 UI 拉取某个 Worker 的执行日志卡片
CREATE INDEX idx_tool_log_session ON tool_execution_log(session_id);
-- 索引：加速通过拦截日志逆向追溯大模型的原始 Prompt 思考
CREATE INDEX idx_tool_log_call_id ON tool_execution_log(tool_call_id);

```

### 三 核心契约：密封时间接口 DbWriteEvent
在 Java 21 中，使用 sealed interface 可以严格限制有哪些事件可以操作数据库，这在底层框架设计中提供了极强的安全性。
```java
package com.yourproject.module6.event;

import com.yourproject.module5.model.UnifiedMessage;

/**
 * 模块 6：数据库写操作事件的基类
 * 采用 sealed interface，明确规定只有以下六种事件允许写库。
 */
public sealed interface DbWriteEvent permits
    InsertTaskSessionEvent,
    UpdateTaskStatusEvent,
    InsertMessageEvent,
    InsertToolExecutionLogEvent,
    DeleteMessagesEvent,
    DeleteTaskSessionEvent {
}

// 具体的事件载体 (使用 Record，天生不可变，极其适合事件驱动)
public record InsertTaskSessionEvent(String sessionId, String parentId, String role, String topic) implements DbWriteEvent {}

public record UpdateTaskStatusEvent(String sessionId, String newStatus, String planJson) implements DbWriteEvent {}

public record InsertMessageEvent(String sessionId, String streamType, int sequenceNum, UnifiedMessage message) implements DbWriteEvent {}

public record InsertToolExecutionLogEvent(String sessionId, String toolCallId, String toolName, String result) implements DbWriteEvent {}

/**
 * 删除消息记录事件
 * 用于用户删除群聊时的级联数据清理，物理销毁指定会话的所有消息记录
 */
public record DeleteMessagesEvent(String sessionId) implements DbWriteEvent {}

/**
 * 删除任务会话记录事件
 * 用于用户删除群聊时的级联数据清理，物理销毁任务会话本身
 */
public record DeleteTaskSessionEvent(String sessionId) implements DbWriteEvent {}
```
### 四 核心引擎：单线程防锁死持久化机制
```java
// 1. 完全对齐文档命名：SingleThreadDbWriter
public class SingleThreadDbWriter {

    // 2. 内部私有化封装文档中的 MemoryWriteQueue
    private final BlockingQueue<DbWriteEvent> memoryWriteQueue = new LinkedBlockingQueue<>();
    private volatile boolean isRunning = true;

    // 对外只提供投递口
    public void submitEvent(DbWriteEvent event) {
        memoryWriteQueue.offer(event);
    }

    // 内部启动唯一的消费者线程
    public void startConsumer() {
        Thread daemonThread = new Thread(() -> {
            while (isRunning) {
                try {
                    // 只有这里能从 memoryWriteQueue 中 take 数据
                    DbWriteEvent event = memoryWriteQueue.take();
                    dispatchAndExecuteSql(event);
                } catch (InterruptedException ex) {
                    Thread.currentThread().interrupt();
                    isRunning = false;
                }
            }
        }, "SQLite-Single-Thread-Writer");
        daemonThread.setDaemon(true);
        daemonThread.start();
    }

    private void dispatchAndExecuteSql(DbWriteEvent event) {
        switch (event) {
            case InsertTaskSessionEvent e -> {
                // 执行 INSERT INTO task_session ...
            }
            case UpdateTaskStatusEvent e -> {
                // 执行 UPDATE task_session SET status = ...
            }
            case InsertMessageEvent e -> {
                // 执行 INSERT INTO chat_message ...
            }
            case InsertToolExecutionLogEvent e -> {
                // 执行 INSERT INTO tool_execution_log ...
            }
            case DeleteMessagesEvent e -> {
                // 执行 DELETE FROM chat_message WHERE session_id = ?
                // 同时级联删除 tool_execution_log 中的相关记录
            }
            case DeleteTaskSessionEvent e -> {
                // 执行 DELETE FROM task_session WHERE id = ?
            }
        }
    }
}
```
### 五 对外暴露的窗口：Repository接口
业务层（如调度器）是绝对不允许直接操作队列的，它们只能调用 Repository。
```java
import com.yourproject.module5.model.UnifiedMessage;
import java.util.List;

public interface ChatMessageRepository {

    // ========== 读操作 (高并发直接查库) ==========

    /**
     * 拉取特定会话在特定数据流下的完整上下文，用于大模型推理前的 Prompt 组装。
     * 必须严格按照 sequence_num 升序排列，防止出现大模型乱序幻觉。
     * 对应表和索引: chat_message 的 idx_chat_msg_seq
     */
    List<UnifiedMessage> fetchContext(String sessionId, String streamType);

    // ========== 写操作 (封装 Event 进队列) ==========

    /**
     * 异步保存单条消息记录。
     * 供 GlobalStreamManager 或 LocalStreamManager 调用。
     */
    void saveMessage(String sessionId, String streamType, int sequenceNum, UnifiedMessage message);
}
```
