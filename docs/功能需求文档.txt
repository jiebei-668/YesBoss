1. 核心定位与多智能体架构
1.1 结果导向的极简体验：系统的根本目的是让用户与 AI Agent 交互解决问题。用户只需下达任务并确认需求，所有的中间执行过程最大化交由 Agent 自动处理。

1.2 主从层级架构 (Master-Worker)： 系统包含一个“总 Agent”和多个“子 Agent”。

1.3 总 Agent 作为中枢，全权负责与用户交互、任务规划、分配任务给子 Agent，并解答子 Agent 在执行中遇到的问题。

1.4 子 Agent 作为执行单元，严格听从总 Agent 的调度。

2. 任务执行与协同流转机制
2.1 事前排雷与固定规划：接收到初始任务后，总 Agent 会先与用户进行彻底的需求确认。确认无误后，总 Agent 必须执行一段固定的“规划流程”（该流程被封装为工具），并通过专门定制的 Prompt 将大任务拆分为细小的子任务下发。

2.2 严格串行与状态隔离：子 Agent 的执行是严格串行的。各个子 Agent 之间完全隔离，不共享任何上下文；子 Agent 所有的上下文信息仅来源于总 Agent 的单向投喂。

2.3 死循环熔断机制：为避免任务卡死，总 Agent 与单一子 Agent 针对某一环节的对话轮数上限被严格固定为 20 轮。

3. 交互体验与 IM 系统集成
3.1 深度接入办公 IM：用户端完全基于 IM 软件操作，系统需集成飞书 (Feishu) 和 Slack。

【Webhook 异步化响应机制】企业级 IM（飞书/Slack）的 Webhook 回调有严格的 ACK 超时限制（通常 3 秒内必须返回 200 OK）。系统必须在 Gateway 层实现真正的异步化：
- WebhookController 收到 payload 后，仅进行快速验签和解析（<100ms）
- 将其封装为内部事件（ImWebhookEvent）并丢入异步线程池（虚拟线程）
- **当前 HTTP 请求必须立即返回 200 OK**（在 3 秒内完成）
- 所有的路由、SessionManager 调用、LLM 推理均在后台虚拟线程中异步执行
- 结果通过 IMMessagePusher 主动推送回 IM 系统（不依赖 Webhook 响应）

【技术实现说明】异步化使用 Java 21+ 的虚拟线程（Virtual Thread），通过 Executors.newVirtualThreadPerTaskExecutor() 实现 WebhookEventExecutor，确保高并发下的资源效率。

3.2 会话与群聊强绑定：一次完整的任务（如分析某代码项目、完成某份报告）即为一次"会话"。每次开启新任务，系统自动新建一个专属群聊。当用户删除该群聊时，此次会话的所有关联数据也会被同步销毁。

3.3 人机回环 (Human-in-the-loop)：执行过程中系统默认静默运转。但当总/子 Agent 遇到敏感操作（触发黑名单）时，总 Agent 会在群内 @ 用户发起提问，系统将无限期挂起（无超时机制），直到用户明确回复。

【技术实现说明】子 Agent 在触发黑名单后会完全释放线程资源并退出，不占用任何系统资源。
当用户回复后，系统会从数据库恢复完整上下文并重新启动子 Agent 线程继续执行。

3.4 极简 UI 与总结卡片：

    执行中：群聊内仅显示用户与总 Agent 的对话，以及极其简短的进度条（仅提示“正在完成 xxx 任务...”）。

    执行后：总 Agent 返回一张任务总结卡片，展示各子任务的完成情况。总 Agent 与子 Agent 在执行过程中的中间对话，默认收纳在卡片的折叠面板中，保持界面清爽。

4. 上下文管理、模型与工具链
4.1 多模型灵活配置：系统需兼容多种大语言模型（Claude、Gemini、GPT、ZhipuGLM）。总 Agent 和子 Agent 支持独立配置不同的 LLM。

4.2 智能上下文切片与压缩：在拆分子任务后，子 Agent 仅获取“之前的部分相关对话”以及“完成当前子任务产生的所有对话”。为了控制 Token 消耗，在单个子任务的会话内部必须实现自动的 Compact（上下文压缩）机制。

4.3 丰富的工具集：总 Agent 和子 Agent 均被赋予强大的工具调用能力，包括但不限于文件操作、Blob 处理、网络搜索，并全面支持各类型的 MCP (Model Context Protocol) 工具。

5. 数据持久化与安全防御
5.1 SQLite 并发持久化：所有的用户与 Agent 对话、Agent 内部对话以及工具调用记录都必须完整保留。底层采用 SQLite 进行本地持久化，并在系统设计上必须重点解决和保障并发写的数据安全。

5.2 命令黑名单沙箱：为防止 Agent 利用文件和系统工具造成破坏，系统需实现基础的安全沙箱环境，目前的核心机制是通过设置"命令黑名单"来拦截高危操作。

**拦截后的用户确认机制：** 当工具调用触发黑名单被拦截后，系统必须立即将任务状态转为 SUSPENDED（挂起），并通过 IM 系统在群内 @ 用户展示拦截详情（包括被拦截的工具名称和参数）。系统将无限期等待用户的明确决策，只有用户本人可以授权继续执行、拒绝操作或提供补充指令。总 Agent（Master）仅作为消息转发的中介，无权代替用户做出决策，也无法自行恢复被挂起的子 Agent（Worker）执行。
