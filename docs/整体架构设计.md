# 系统整体架构设计文档 (High-Level Architecture) - V5.0

## 1. 系统定位与核心目标

本系统是一个极简体验、结果导向的多智能体（Multi-Agent）任务编排平台。系统采用主从（Master-Worker）层级架构，深度接入办公 IM（如飞书/Slack）。其核心目标是通过“总 Agent 全局探索与规划”结合“子 Agent 局部隔离执行”的模式，在安全受控的本地环境中，以最大化自动化的方式解决复杂的工程与业务任务。

## 2. 核心架构设计理念 (Core Architectural Principles)

为确保系统在处理复杂长链条任务时的绝对稳定性与高扩展性，本架构确立了两大核心设计基石：

### 2.1 混合驱动模式 (Hybrid Engine)

系统严格界定了“大模型自由发挥”与“框架强制管控”的边界：

* **业务解题层（动态自主驱动）：** Agent 在执行“环境探索”或“代码修改”时，处于纯正的 ReAct 循环中。大模型拥有完全的自主权，根据现状动态决定调用哪些工具、如何分析报错。
* **生命周期与上下文管理层（框架固定流程）：** 在发生“跨 Agent 任务交接”（向下注入与向上收敛）以及“安全拦截挂起”时，系统剥夺大模型的自由度，强制切入底层写死的框架逻辑。避免了规划丢失、上下文污染或因 Agent 熔断导致的系统卡死。

### 2.2 统一工具抽象层 (Unified Tool Abstraction)

系统对底层能力实行彻底的适配器模式（Adapter Pattern）封装：

* 原生函数（如本地文件读写）与外部 MCP（Model Context Protocol）工具，在注册入网时均被包装为极其统一的标准 Tool 接口。
* 对上层 Agent 而言，屏蔽了所有底层协议差异；对底层安全沙箱而言，只需在调用有风险的工具前使用死逻辑判断(TODO后续可能会改成切面），极大降低了安全防线的复杂度和遗漏风险。

## 3. 整体架构分层与组件 (Architecture Layers)

系统采用“业务编排 + 共享核心引擎”的矩阵式设计。

### 3.1 接入路由层 (Gateway Layer)

* **IM Webhook 路由：** 实时捕获并解析办公软件的群聊事件及@事件。
* **会话生命周期管理器：** 负责“一任务一群聊”的创建，以及群聊解散时的级联数据物理销毁。
* **极简 UI 渲染器：** 将系统内部复杂的推演流和多轮对话，转化为对用户友好的进度条或带有折叠面板的“任务总结卡片 (Summary Card)”。

### 3.2 业务编排层 (Agent Orchestration Layer)

系统的业务中枢，包含两个独立运转但互相配合的角色。

* **中枢总 Agent (Master)：全局管控者**
  * **核心流转：** 需求澄清 -> 环境探索 (自主调工具) -> 制定规划 -> 任务分发 -> 异常拦截与人工回环处理。(TODO这个异常拦截和人工回环应该是固定逻辑，而不是主agent的业务)
  * **权限边界：** 拥有全局流上下文的读写权；仅被授权调用只读/探索型工具（如查看目录、读取核心源码）。


* **执行子 Agent (Worker)：隔离攻坚者**
  * **核心流转：** 接收明确目标 -> 微观 ReAct 循环 (自主调工具) -> 生成执行报告 -> 触发 20 轮死循环强制熔断。
  * **权限边界：** 仅拥有局部隔离流上下文的读写权；被授权调用读写执行全量工具（受严格沙箱管控）。



### 3.3 共享核心引擎层 (Shared Core Engine Layer)

系统的基础设施，提供上下文路由与安全调度。

* **双流上下文管理器 (Dual-Stream Context Manager)：**
  * **全局流 (Global Stream)：** 维护 Master 的全局约束、项目探索摘要、子任务队列。绝对隔离底层执行的"脏数据"(细节数据)。
  * **局部流 (Local Stream)：** 维护 Worker 的单次任务目标及试错报错明细。各 Worker 之间物理隔离，绝不交叉(目前绝不主动交叉，TODO后续主agent可能会将其他子agent的必要的信息作为该子agent的部分上下文）。


* **统一工具链与安全沙箱 (Unified Toolchain & Sandbox)：**
  * **全量工具管控：** 统一管理原生 Tool 与 MCP 适配器 Tool。
  * **分级权限控制 (RBAC)：** 根据 Master 与 Worker 的身份动态应用不同策略。
  * **高危命令拦截器：** 基于黑名单拦截破坏性操作。触发时立即向上抛出挂起 (Suspend) 信号，阻断执行并请求人工介入。逻辑上子agent需要中转主agent，因为会给主agent的上下文添加这个询问的信息，实现上是跨过主agent的llm的，直接查到该会话对应群聊然后发起提问。

  * **用户确认的强制性：** 被拦截后，系统必须通过总 Agent（Master）向用户发起确认请求，只有用户本人可通过 IM 界面授权继续或拒绝。总 Agent 严禁自行决策或绕过用户直接恢复子 Agent（Worker）的执行，确保危险操作的最终控制权始终掌握在人类手中。



### 3.4 基础设施与持久化层 (Infrastructure Layer)

* **多模型路由网关：** 屏蔽模型服务商差异，支持为 Master 规划、Worker 执行、框架级总结分配不同的 LLM 实例（平衡成本与智力）。
* **SQLite 并发控制器：**
* 采用异步内存无锁队列接收所有模块的对话、内部状态与工具调用日志。
* 依靠后台单一消费者线程顺序批量落盘，从物理层面根除文件型数据库的高并发写冲突问题。
* **级联数据销毁机制：** 当用户删除群聊时，通过删除事件（DeleteMessagesEvent、DeleteTaskSessionEvent）触发级联清理，物理销毁所有关联的任务会话、消息记录和工具执行日志，确保用户隐私和存储空间的有效管理。



## 4. 关键流转机制：跨 Agent 状态交接

框架级固定流程，不依赖当前 Agent 的自主调用：

* **向下注入 (Top-Down Injection)：**
* Master 产出结构化任务后，框架接管。框架提取全局流中的规则，通过硬编码模板与任务目标拼装，作为初始系统提示词静默注入给新建的 Worker。


* **向上收敛 (Bottom-Up Condensation)：**
* Worker 成功完成或触发熔断后，框架接管其庞大的局部流历史。框架开启独立的全新 Context，调用和 Worker 相同的 LLM，注入"系统级日志压缩规则"，将其提纯为简短的执行报告（Observation），最后合并回 Master 的全局流。



## 5. 核心业务流程：人机回环 (Human-in-the-Loop)

人机回环是系统安全防护的最后一道防线，当 Worker 尝试调用黑名单工具时触发。系统通过5步完美闭环实现"拦截-挂起-求助-决策-缝合"的全流程控制。

### 5.1 步骤1：底层拦截与异常抛出 (Intercept)

**场景：** Worker 在 ReAct 循环中决定调用某个工具

**触发点：** `模块4::SandboxInterceptor::preCheck()` 在工具执行前进行黑名单校验

**核心动作：**
- 拦截器检查工具名称和参数是否匹配黑名单规则
- 若命中黑名单，立即阻断执行
- 向上抛出 `SuspendExecutionException` 异常，携带三个关键信息：
  - `toolCallId`：大模型生成的调用ID（因果纽带，用于后续缝合上下文）
  - `toolName`：被拦截的工具名称
  - `argumentsJson`：工具参数的完整JSON

**设计意图：** 在最早的切入点拦截危险操作，并完整保留调用现场，为后续恢复提供必要数据。

---

### 5.2 步骤2：状态机挂起 (Suspend)

**场景：** Worker 的执行逻辑捕获到拦截异常

**处理点：** `模块2::SuspendResumeEngine::suspendForApproval()`

**核心动作：**
1. **状态转换：** 将 Worker 的状态从 `RUNNING` 切换为 `SUSPENDED`
2. **现场持久化：** 将 `toolCallId`、`toolName`、`argumentsJson` 存入数据库，供唤醒时恢复
3. **线程完全退出：** Worker 的 `run()` 方法捕获异常后**正常退出**，释放所有线程和内存资源。系统进入无限期等待，期间仅依赖数据库持久化状态维持，不占用任何线程资源

**设计意图：** 通过状态机强制暂停执行，防止危险操作继续，同时完整保存调用现场以便后续恢复。

---

### 5.3 步骤3：跨层求助与物理触达 (Spoof & Push)

**场景：** Worker 已挂起，需要向用户发起确认请求

**核心设计：** 越过 Master 的 LLM，直接在 Master 的全局流中伪造系统提问

**处理点：** `模块3::GlobalStreamManager::appendSystemMessage()` + `模块1::IMMessagePusher`

**核心动作：**
1. **跨Agent伪造：** 引擎查找 Worker 的父级 Master SessionId，直接向 Master 的全局流插入一条系统消息（不经过 Master 的 LLM），内容格式：
   ```
   【系统拦截】子Agent尝试执行高危操作：
   工具：rm_file
   参数：{"path": "/etc/passwd"}

   请确认是否授权？
   ```
2. **物理触达：** 通过 `IMMessagePusher` 将渲染好的交互卡片（包含"授权"和"拒绝"按钮）推送到用户的群聊，并 @用户

**设计意图：**
- **越过 Master LLM：** Master 仅作为消息转发中介，严禁自行决策，确保最终控制权在人类手中
- **全局流记录：** 将拦截事件记入 Master 的全局流，保持会话历史的完整性
- **主动推送：** 不等待用户主动查询，而是通过 IM 主动触达，确保用户及时知晓

---

### 5.4 步骤4：人类决断与路由回调 (Human Decision)

**场景：** 用户在群聊中点击了卡片按钮

**入口点：** `模块1::WebhookController` 接收 IM 系统的回调事件

**核心动作：**
1. 解析用户决策（授权/拒绝/补充指令）
2. 提取关键字段：`workerSessionId`、`toolCallId`、`isApproved`、`humanFeedback`
3. 路由到 `模块2::SuspendResumeEngine::resume()` 方法

**用户决策的三种情况：**
- **授权**：`isApproved=true`，继续执行工具
- **拒绝**：`isApproved=false`，终止该次调用
- **补充指令**：`isApproved=true` + `humanFeedback`，按用户的建议方式执行

**设计意图：** 将用户的真实决策通过网关层路由回系统内部，唤醒挂起的 Worker。

---

### 5.5 步骤5：结果落定与记忆缝合 (Execute/Forge & Resume)【核心】

**场景：** 挂起引擎接收到用户的明确决策，需要生成 ToolResult 并唤醒 Worker

**处理点：** `模块2::SuspendResumeEngine::resume()`

**核心动作：**

**分支A：用户同意授权 (isApproved=true)**
1. 从数据库恢复拦截现场的 `toolName` 和 `argumentsJson`
2. 获取工具实例，**绕过沙箱黑名单检查**（`executeWithBypass()` 模式），直接执行工具
3. 将真实的执行结果封装为 `UnifiedMessage.ofToolResult(toolCallId, realResult, isError=false)`
4. 注入 Worker 的局部流：`LocalStreamManager.appendToolResult(workerSessionId, toolResultMsg)`
5. 状态切回 `RUNNING`，创建**新线程**重新调用 `WorkerRunner.run(sessionId)`

**分支B：用户拒绝授权 (isApproved=false)**
1. **不执行任何底层工具**
2. 伪造一段错误文本：
   ```
   工具调用失败，安全沙箱已拦截：用户拒绝授权，没有权限执行此操作。
   用户指导意见：<用户的补充意见>
   ```
3. 封装为 `UnifiedMessage.ofToolResult(toolCallId, forgedError, isError=true)`
4. 注入 Worker 的局部流
5. 状态切回 `RUNNING`，创建**新线程**重新调用 `WorkerRunner.run(sessionId)`

**设计意图：**
- **同意时**：绕过沙箱执行，返回真实数据 → 大模型拿到结果继续干活
- **拒绝时**：伪造"权限不足"错误 → 大模型理解失败原因，尝试其他非黑名单工具
- **状态保持 RUNNING**：不立即结束任务，给模型探索替代方案的机会（若20轮后仍无法完成，再触发熔断转为 FAILED）
- **ToolResult 缝合**：通过 `toolCallId` 关联，确保大模型准确理解是哪次调用被处理

---

### 5.6 完整闭环的价值

这5步流程构成了一个完美的闭环：

1. **安全拦截**：在最早的切入点阻断危险
2. **状态持久**：完整保存调用现场
3. **人类介入**：越过 AI 直接触达人类决策者
4. **意图路由**：将人类决策准确注入系统
5. **记忆缝合**：无论授权还是拒绝，都生成规范的 ToolResult，让大模型理解发生了什么

**核心原则：**
- Master 严禁自行决策，只能作为消息转发中介
- 用户拥有最终控制权，系统无限期等待明确回复
- 拒绝后不立即终止任务，给模型探索替代方案的机会
- ToolResult 的因果纽带（toolCallId）始终完整，保证上下文可追溯



## 6. 核心业务数据流转 (Data Flow Example)

1. **触发：** 用户在 IM 下达重构任务，Gateway 创建专属群聊。
2. **探索 (Explore)：** Master 唤醒，进入 ReAct 循环，自主调用只读工具扫描目录，框架将核心发现记入全局流。
3. **规划 (Plan)：** Master 输出子任务拆解（如：“重构 `tui` 模块”）。
4. **注入 (Inject)：** 框架拦截任务，纯代码拼装全局 Java 并发规范与当前目标，下发给 Worker A。
5. **执行 (Execute)：** Worker A 在沙箱内自主调用读写及终端编译工具。期间的所有编译报错均留存在隔离的局部流中。
6. **拦截 (可选)：** Worker A 尝试越权操作，沙箱拦截器阻断执行。状态机持久化并挂起，Master 在群内@用户求助。
7. **收敛 (Condense)：** Worker A 历经多轮尝试完成重构。框架提取其冗长的局部流，通过独立 LLM 请求压缩为一句话报告，合并回 Master 的全局流。
8. **落盘：** 全程产生的全局日志与被收敛掉的局部明细日志，均通过异步队列平滑写入 SQLite 数据库。
