# YesBoss记忆持久化模块 v3.0 - 完整接口设计

## 概述

本文档定义了YesBoss记忆持久化模块的完整接口设计，采用三层架构（Resource、Snippet、Preference），专注于对话形式的记忆处理。支持两种向量存储后端（sqlite-vec和PostgreSQL+pgvector），通过配置文件灵活切换。

**版本变化：**
- v1.0 → v2.0：移除多模态支持，专注conversation
- 简化接口设计，聚焦核心功能
- 修正方法名错误
- 补充缺失的关键接口

## 1. VectorStore向量存储抽象层

### 1.0 架构设计理念

本模块采用**灵活的向量存储架构**，支持多种部署模式以适应不同的场景需求。

#### 推荐架构：单库模式

**最佳实践：将关系数据和向量数据存储在同一个数据库中**

```
┌─────────────────────────────────────────────────┐
│         PostgreSQL 数据库                         │
│  ├─ resources 表（content, abstract）            │
│  ├─ snippets 表（summary, memory_type）          │
│  ├─ preferences 表（name, summary）              │
│  └─ 向量索引（使用 pgvector 扩展）               │
│      ├─ resource_index                           │
│      ├─ snippet_index                            │
│      └─ preference_index                         │
└─────────────────────────────────────────────────┘

或

┌─────────────────────────────────────────────────┐
│         SQLite 数据库                             │
│  ├─ resources 表                                │
│  ├─ snippets 表                                 │
│  ├─ preferences 表                              │
│  └─ 向量索引（使用 sqlite-vec 扩展）             │
└─────────────────────────────────────────────────┘
```

**优势：**
- ✅ **事务一致性**：关系数据和向量在同一事务中更新，保证数据一致性
- ✅ **简单部署**：只需维护一个数据库实例
- ✅ **减少网络开销**：无需跨数据库查询
- ✅ **备份恢复简单**：统一的数据备份策略
- ✅ **性能优异**：利用数据库的原生向量索引能力

#### 可选架构：分离存储模式

**理论支持：将关系数据库和向量数据库分离**

```
┌──────────────────────────┐  ┌──────────────────────────┐
│   关系数据库               │  │   向量数据库               │
│   (PostgreSQL/MySQL)     │  │   (PostgreSQL+pgvector)  │
│                          │  │                          │
│  ├─ resources 表         │  │  ├─ resource_index        │
│  ├─ snippets 表          │  │  ├─ snippet_index         │
│  └─ preferences 表       │  │  └─ preference_index      │
└──────────────────────────┘  └──────────────────────────┘
          │                              │
          └─────── 通过 metadata 关联 ──────┘
```

**使用场景：**
- 现有系统已有独立的关系数据库和向量数据库
- 需要使用专门的向量数据库服务（如Milvus、Qdrant等）
- 不同团队维护不同的数据库组件

**挑战：**
- ⚠️ **事务协调**：需要额外的机制保证跨库事务一致性
- ⚠️ **数据同步**：删除关系数据时需要同步删除向量数据
- ⚠️ **复杂度增加**：需要处理网络分区、故障转移等问题
- ⚠️ **性能开销**：跨数据库查询增加网络延迟

#### VectorStore抽象层的灵活性

```java
// 配置示例1：单库模式（推荐）
VectorStoreConfig config = new VectorStoreConfig();
config.setType(VectorStoreType.POSTGRESQL_PGVECTOR);
config.setConnectionString("jdbc:postgresql://localhost:5432/memory_db");
// VectorStore和Repository使用同一个数据库

// 配置示例2：分离模式（可选）
VectorStoreConfig config = new VectorStoreConfig();
config.setType(VectorStoreType.POSTGRESQL_PGVECTOR);
config.setConnectionString("jdbc:postgresql://vector-host:5432/vector_db");
// VectorStore使用独立的向量数据库
```

**设计原则：**
1. **默认单库**：推荐使用单库模式，除非有明确的需求
2. **接口统一**：无论哪种模式，VectorStore接口保持一致
3. **配置驱动**：通过配置文件灵活切换，无需修改代码
4. **向后兼容**：未来可以轻松切换存储模式

#### 与memU的对比

| 特性 | memU | v3.0（YesBoss） |
|------|------|-----------------|
| **推荐架构** | 单库 | 单库 |
| **PostgreSQL** | ✅ pgvector扩展 | ✅ pgvector扩展 |
| **SQLite** | JSON存储embedding | ✅ sqlite-vec扩展（性能更优）|
| **分离存储** | ❌ 不支持 | ✅ 理论支持（不推荐）|
| **抽象层** | 直接使用ORM | VectorStore抽象层 |

v3.0在memU的基础上增强了灵活性，同时保持了单库模式的简洁性。

package tech.yesboss.memory.vector;

import java.util.List;
import java.util.Map;

/**
 * 向量存储抽象接口
 * 支持多种向量数据库后端（sqlite-vec, PostgreSQL+pgvector）
 */
public interface VectorStore {

    /**
     * 创建向量索引
     * @param indexName 索引名称
     * @param dimensions 向量维度
     */
    void createIndex(String indexName, int dimensions);

    /**
     * 插入向量数据
     * @param indexName 索引名称
     * @param id 数据ID
     * @param vector 向量数据
     * @param metadata 元数据
     */
    void insert(String indexName, String id, float[] vector, Map<String, Object> metadata);

    /**
     * 批量插入向量数据
     * @param indexName 索引名称
     * @param vectorDataList 向量数据列表
     */
    void bulkInsert(String indexName, List<VectorData> vectorDataList);

    /**
     * 向量相似度搜索
     * @param indexName 索引名称
     * @param queryVector 查询向量
     * @param topK 返回前K个结果
     * @return 搜索结果列表
     */
    List<VectorSearchResult> search(String indexName, float[] queryVector, int topK);

    /**
     * 带过滤条件的向量搜索
     * @param indexName 索引名称
     * @param queryVector 查询向量
     * @param topK 返回前K个结果
     * @param filters 过滤条件
     * @return 搜索结果列表
     */
    List<VectorSearchResult> search(String indexName, float[] queryVector, int topK, Map<String, Object> filters);

    /**
     * 更新向量数据
     * @param indexName 索引名称
     * @param id 数据ID
     * @param vector 向量数据
     * @param metadata 元数据
     */
    void update(String indexName, String id, float[] vector, Map<String, Object> metadata);

    /**
     * 删除向量数据
     * @param indexName 索引名称
     * @param id 数据ID
     */
    void delete(String indexName, String id);

    /**
     * 批量删除向量数据
     * @param indexName 索引名称
     * @param ids 数据ID列表
     */
    void bulkDelete(String indexName, List<String> ids);

    /**
     * 获取向量数据
     * @param indexName 索引名称
     * @param id 数据ID
     * @return 向量数据
     */
    VectorData get(String indexName, String id);

    /**
     * 获取索引信息
     * @param indexName 索引名称
     * @return 索引信息
     */
    IndexInfo getIndexInfo(String indexName);

    /**
     * 检查索引是否存在
     * @param indexName 索引名称
     * @return 是否存在
     */
    boolean indexExists(String indexName);

    /**
     * 删除索引
     * @param indexName 索引名称
     */
    void dropIndex(String indexName);
    /**
     * 批量更新Resource的向量
     * @param resources Resource列表
     */
    void batchUpdateResourceEmbeddings(List<Resource> resources);

    /**
     * 批量更新Snippet的向量
     * @param snippets Snippet列表
     */
    void batchUpdateSnippetEmbeddings(List<Snippet> snippets);

    /**
     * 批量更新Preference的向量
     * @param preferences Preference列表
     */

}

/**
 * 向量存储类型枚举
 */
public enum VectorStoreType {
    SQLITE_VEC("sqlite-vec"),
    POSTGRESQL_PGVECTOR("postgresql-pgvector");

    private final String value;

    VectorStoreType(String value) {
        this.value = value;
    }

    public String getValue() {
        return value;
    }

    public static VectorStoreType fromValue(String value) {
        for (VectorStoreType type : values()) {
            if (type.value.equals(value)) {
                return type;
            }
        }
        throw new IllegalArgumentException("Unknown vector store type: " + value);
    }

    public void setEmbedding(float[] embedding) { this.embedding = embedding; }

    public long getTimestamp() { return timestamp; }
    public void setTimestamp(long timestamp) { this.timestamp = timestamp; }

    public int getMessageCount() { return messageCount; }
    public void setMessageCount(int messageCount) { this.messageCount = messageCount; }

    public Map<String, Object> getMetadata() { return metadata; }
    public void setMetadata(Map<String, Object> metadata) { this.metadata = metadata; }

    public void setDeleted(boolean deleted) { this.deleted = deleted; }

    public void setCreatedAt(LocalDateTime createdAt) { this.createdAt = createdAt; }

            resource.setId(id);
            return this;
        }

        public Builder conversationId(String conversationId) {
            resource.setConversationId(conversationId);
            return this;

            resource.setSessionId(sessionId);
            return this;

            resource.setContent(content);
            return this;

            resource.setAbstract(abs);
            return this;
        }

            resource.setEmbedding(embedding);
            return this;
        }

        public Builder timestamp(long timestamp) {
            resource.setTimestamp(timestamp);
            return this;
        }

        public Builder messageCount(int count) {
            resource.setMessageCount(count);
            return this;
        }

        public Builder metadata(Map<String, Object> metadata) {
            resource.setMetadata(metadata);
            return this;
        }

        public Builder deleted(boolean deleted) {
            resource.setDeleted(deleted);
            return this;
        }

        public Resource build() {
            // 设置默认值
            if (resource.getCreatedAt() == null) {
                resource.setCreatedAt(LocalDateTime.now());
            }
            if (resource.getUpdatedAt() == null) {
                resource.setUpdatedAt(LocalDateTime.now());
            }
            // 确保deleted字段有默认值
            resource.setDeleted(false);
            return resource;
        }
    }
}

package tech.yesboss.memory.processor;

import tech.yesboss.memory.model.Snippet;
import tech.yesboss.memory.model.ConversationSegment;
import java.util.List;

/**
 * 内容处理服务
 * 负责为对话生成abstract和summary
 */
public interface ContentProcessor {

    /**
     * 分析并分割对话
     *
     * LLM分析完整对话内容，按照主题自动分割成多个segments。
     *
     * @param conversationContent 完整对话内容（已格式化，每条消息一行）
     * @return 对话分段列表，按时间顺序排列
     */
    List<ConversationSegment> segmentConversation(String conversationContent);

    /**
     * 生成分段摘要
     *
     * 为单个对话分段生成1-2句简短的abstract摘要。
     *
     * @param segmentContent 分段内容
     * @return 分段摘要
     */
    String generateSegmentAbstract(String segmentContent);

    /**
     * 生成对话描述
     * @param content 对话内容
     * @return 对话描述
     * @deprecated 使用 segmentConversation + generateSegmentAbstract 代替
     */
    @Deprecated
    String generateAbstract(String content);

    /**
     * 从Resource中提取结构化记忆（对应memU的generate_entries）
     *
     * LLM从Resource的完整内容中提取特定类型的结构化记忆片段。
     * 每个片段是一个独立的、有意义的记忆单元（如人物档案、事件、知识等）。
     *
     * @param resourceContent Resource的完整内容（对话分段、文档等）
     * @param memoryType 记忆类型（profile, event, knowledge, behavior等）
     * @return 提取的结构化记忆列表，每个元素是一个结构化记忆片段
     */
    List<String> extractStructuredMemories(String resourceContent, MemoryType memoryType);

    /**
     * 识别Snippet属于哪些Preference类别（对应memU的classify_memory_item）
     *
     * LLM分析Snippet的内容，判断它应该关联到哪些现有的、硬编码的Preference主题。
     * 一个Snippet可以关联多个Preference（many-to-many关系）。
     *
     * 注意：Preference是硬编码的，不会动态创建。
     *
     * @param snippetSummary Snippet的摘要内容
     * @param existingPreferences 现有的、硬编码的Preference列表（包含id、name、summary等）
     * @return Preference ID列表（可能为空，表示不属于任何现有Preference）
     */
    List<String> identifyPreferencesForSnippet(String snippetSummary, List<Preference> existingPreferences);

    /**
     * 更新Preference摘要
     *
     * 使用现有的Preference summary和新的Snippets生成更新后的summary。
     * LLM会执行merge操作：保留历史信息，添加新信息。
     *

### 6.1 MemoryManager设计说明

#### 6.1.1 架构定位

MemoryManager位于Repository层和Service层之间，负责：

1. **三层协调**：管理Resource、Snippet、Preference之间的关联关系
2. **事务管理**：确保跨层数据操作的一致性
3. **批量优化**：提供批量操作接口提升性能
4. **向量同步**：协调向量化操作与数据存储

#### 6.1.2 方法设计原则

1. **单/批量方法成对**：每个save方法都有对应的批量版本
   - `saveResource()` ↔ `saveResources()`
   - `saveSnippet()` ↔ `saveSnippets()`
   - `savePreference()` (创建 Preference)

2. **创建/更新分离**：Preference使用专门的update方法更新summary和statistics

3. **向量延迟生成**：数据创建和向量化分离，支持批量优化

#### 6.1.3 与其他接口的关系

```
MemoryService (业务编排层)
       ↓
MemoryManager (三层协调层)
       ↓
    ┌───────┬──────────┬──────────┐
    ↓       ↓          ↓          ↓
ResourceRepo SnippetRepo PreferenceRepo VectorStore
(数据持久化)    (向量索引)
```

#### 6.1.4 使用场景

| 场景 | 使用的方法 |
|------|-----------|
| 实时记忆提取 | `saveResource()`, `saveSnippet()`, `savePreference()` |
| 批量记忆处理 | `saveResources()`, `saveSnippets()` |
| 更新Preference | `updatePreferenceSummary()` |
| 批量向量化 | `batchUpdateResourceEmbeddings()`, `batchUpdateSnippetEmbeddings()`, `batchUpdatePreferenceEmbeddings()` |

     * 避免重复处理。
     *

     * 统计未处理数量

    long countUnprocessed();

     * @param messageIds 对话ID列表
     */

**与现有代码的集成：**

2. **实现方式**：可以创建一个wrapper类或直接在现有实现类中实现这些方法
3. **异步处理**：markAsProcessed应该异步执行，不阻塞主流程
